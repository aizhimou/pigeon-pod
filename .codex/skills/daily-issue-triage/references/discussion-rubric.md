# Discussion Rubric

Use this when an issue is mainly about product boundaries, usage guidance, workflow expectations, or integration discussion rather than a clearly scoped bug or feature request.

## Output Shape

- Discussion summary
- What the user is really asking
- Product-boundary assessment
- Repo or architecture evidence
- Recommended maintainer stance
- Draft reply
- Recommendation

## Recommendation Values

- `reply_only`
- `reply_and_monitor`
- `promote_to_requirement`
- `close_candidate`

## Heuristics

- Use `reply_only` when the best response is guidance or boundary clarification.
- Use `reply_and_monitor` when the discussion may later become actionable but is not there yet.
- Use `promote_to_requirement` only when the thread has converged on a concrete product change worth planning.
- Use `close_candidate` when the issue is resolved by explanation, no longer actionable, or outside product scope.

## Reply Style

- Be technically specific.
- Be honest about product boundaries.
- Do not over-promise implementation.
- Prefer one clear recommendation over many optional ideas.
